# API Documentation

## Overview

The prediction API serves precomputed term deposit subscription propensity predictions by `user_id`.

The API does not perform realtime model inference. Predictions are generated ahead of time by the PySpark pipeline and stored in a prediction output table. The API looks up a requested `user_id` and returns the matching prediction result.

Current MVP data source:

```text
api_project/data/mock_predictions_500k.parquet
```

The current API serves mock prediction output for lookup and load-test readiness. It does not yet serve the Logistic Regression output from the PySpark pipeline.

## Field Definitions

Successful response fields:

| Field | Type | Description |
| --- | --- | --- |
| `user_id` | string | Client identifier used for lookup. |
| `prediction_score` | float | Probability-like score for term deposit subscription. |
| `prediction_label` | int | Predicted class label, where `1` means likely subscriber and `0` means not likely subscriber. |
| `model_version` | string | Model version that produced the prediction. |
| `scored_at` | timestamp | Time when the prediction was generated. |
| `request_id` | string | Unique ID for the API request. |
| `trace_id` | string | Trace identifier for request observability. |
| `timestamp` | timestamp | Time when the API response was generated. |

Error response fields:

| Field | Type | Description |
| --- | --- | --- |
| `error` | string | Machine-readable error code. |
| `message` | string | Human-readable error message. |
| `request_id` | string | Unique ID for the API request. |
| `trace_id` | string | Trace identifier for request observability. |
| `timestamp` | timestamp | Time when the API response was generated. |

## Prediction Output Table Schema

The API reads from a prediction output table generated by the PySpark pipeline.

| Field | Type | Description |
| --- | --- | --- |
| `user_id` | string | Stable synthetic or generated client identifier. |
| `prediction_score` | float | Probability-like score for term deposit subscription. |
| `prediction_label` | int | Predicted label, where `1` means likely subscriber and `0` means not likely subscriber. |
| `model_version` | string | Version identifier for the model that produced the score. |
| `scored_at` | timestamp | Time when the row was scored. |
| `source` | string | Source category, for example `real`, `mock`, or `expanded`. |

For API and load testing, this output may be expanded or mocked to approximately 500k rows using the same schema. Mock or expanded rows must be labeled with `source` and must not be described as real raw training users.

## Cache Behavior

The MVP cache is an in-memory cache for repeated `user_id` lookups.

Cache hit:

- The requested `user_id` already exists in memory.
- The API can return the cached prediction response without reading from the backing prediction data source.

Cache miss:

- The requested `user_id` is not in memory.
- The API reads from the prediction data source.
- If a prediction is found, the API may store it in memory for future requests.

Redis can be considered later as a more production-like cache, but it is not required for the MVP.

Current implementation:

- Prediction rows are loaded from Parquet into an in-memory dictionary.
- A small LRU-style in-memory cache stores repeated lookup responses by `user_id`.
- `/health` exposes `cache_size`, `cache_hits`, and `cache_misses`.

## Functional Test Plan

Functional tests should cover:

- Existing `user_id` returns HTTP 200.
- Missing `user_id` returns HTTP 404.
- Server failure returns HTTP 500 with the documented error response shape.
- Response includes `request_id`, `trace_id`, and `timestamp`.
- Response field types match the API contract.
- Cache hit and cache miss behavior is verified if cache metrics or logs are available.

## Load Test Plan

Load testing should simulate expected or normal API traffic against the lookup endpoint.

Recommended scenarios:

- Repeated lookups for a small set of common `user_id` values.
- Broad lookups across many `user_id` values.
- Mixed cache-hit and cache-miss traffic.
- Testing against an approximately 500k-row prediction output dataset if available.

The 500k-row dataset may be expanded or mocked for API testing. It must not be claimed as 500k real raw training users unless that is true.

## Stress Test Plan

Stress testing should increase traffic beyond expected load until performance degrades.

The test should observe:

- When P95/P99 latency becomes unacceptable.
- When error rate increases.
- When throughput stops improving.
- Whether cache behavior changes under pressure.
- Whether the API remains stable during traffic spikes.

## Performance Metrics to Collect

- Latency.
- Throughput/RPS.
- Error rate.
- P50 latency.
- P95 latency.
- P99 latency.
- Cache hit rate if available.

## PII and Privacy Notes

- `user_id` can be sensitive or linkable in real systems.
- `prediction_score` can be sensitive depending on the business use case.
- Avoid logging raw PII unnecessarily.
- Use synthetic or anonymized IDs in demos.
- Document access-control assumptions before treating the API as production-like.
- Do not commit raw sensitive data.
- Do not expose prediction output publicly without a clear access-control decision.
